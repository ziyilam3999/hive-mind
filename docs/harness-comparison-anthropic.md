# Hive Mind vs Anthropic Harness: Deep Dive Analysis

## Source References
- [Harness design for long-running application development](https://www.anthropic.com/engineering/harness-design-long-running-apps)
- [Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)

---

## 1. Context Management: Full Context Reset

### Anthropic's Approach
- **Context resets** between sprints -- each new agent starts with a clean slate
- Carries over a **handoff artifact** with just enough state (not the full history)
- Found that **compaction alone wasn't enough** -- Claude Sonnet 4.5 exhibited "context anxiety" even with summarized history
- A reset provides a clean slate at the cost of needing a handoff artifact with enough state
- Trade-offs: introduces orchestration complexity, token overhead, and latency

### Hive Mind's Current Approach
- **Full memory accumulation** -- memory.md grows throughout the pipeline, passed to all downstream agents
- No truncation of memory between agents or stages
- Stage-gated checkpoints (NORMALIZE -> SPEC -> PLAN -> EXECUTE -> REPORT) but no context reset between them
- Within SPEC: researcher -> feature-spec-drafter -> critic all receive the SAME full memory
- Role report contents truncated at 2000 words before inclusion in prompts, but main memory never truncated

### Recommendation: Implement Context Resets

**Where to reset:**
1. Between major stages (SPEC -> PLAN -> EXECUTE -> REPORT)
2. Between execution waves within EXECUTE stage

**How to implement:**
- At stage boundaries: generate a **stage handoff artifact** (structured summary of prior stage output + key decisions)
- Between waves: generate a **wave handoff artifact** (completed stories, remaining work, key learnings from last wave)
- Keep full artifacts on disk for audit, but only pass the handoff summary to the next agent
- This prevents "context anxiety" where agents see a massive history and either panic or declare victory prematurely

**Justification:**
- Anthropic found context anxiety is real even with frontier models
- Hive Mind's memory.md grows unboundedly -- a 20-story pipeline could accumulate massive context
- Clean slate + structured handoff = more focused agent behavior per stage

## 2. Sprint Contract / GAN-Inspired Negotiation

### Anthropic's Approach
- Generator and Evaluator **negotiate a sprint contract** before coding begins
- Contract defines: specific implementation details + testable behaviors that will be verified
- This alignment step prevents "moving goalposts" -- evaluator can only judge against agreed criteria
- The app showed more polish and smoothness than solo runs

### Hive Mind's Current Approach
- **Fixed pipeline, no negotiation** -- plan is finalized before ANY execution agent runs
- AC/EC are generated by separate agents (AC-generator, EC-generator) but the implementer has no input
- Step file is canonical: "STEP-FILE-IS-CANONICAL: the step file is single source of truth"
- No back-and-forth between implementer and tester about what "done" means

### Recommendation: Add Lightweight Contract Negotiation

**Where to apply:**
- Before each story's BUILD phase, have the implementer review ACs/ECs and flag any that are ambiguous or untestable
- If flags raised, run a quick AC/EC refinement loop (not full re-planning)

**Why not full GAN negotiation:**
- Hive Mind already has binary AC/EC evaluation (no subjectivity)
- Full negotiation adds cost and latency per story
- But a quick "review + refine" pass would catch bad ACs before wasting a BUILD attempt

## 3. Self-Evaluation Bias

### Anthropic's Problem
- Agents confidently praise their own mediocre output
- Self-evaluation bias is amplified in subjective tasks like design quality
- Solution: separate Evaluator agent that never saw the generation process

### Hive Mind's Current Approach
- **Already solved for binary criteria** -- tester/evaluator agents are independent from implementer
- Diagnostician + fixer are a pair (not self-review)
- SPEC uses 3-way loop: drafter -> critic (doesn't see research report) -> corrector
- Compliance reviewer is independent from implementer

### Where Hive Mind Could Improve
- **REPORT stage**: reporter agent consolidates data but could be biased toward positive framing
- **Learner agent**: captures "what worked/failed" but may over-emphasize successes
- Consider: independent "devil's advocate" agent that specifically looks for gaps in the report

## 4. Spec Granularity

### Anthropic's Approach
- Planner **deliberately avoids specifying granular technical details**
- Errors in the spec cascade into downstream implementation
- Planner constrains agents on **deliverables** and lets them figure out the path
- Example: expanded a one-sentence prompt into a 16-feature spec across ten sprints

### Hive Mind's Current Approach
- SPEC stays relatively high-level (section-based references like "section 3.1")
- AC/EC generation delegated to downstream agents, not pre-generated in SPEC
- PLAN produces story skeletons with GOAL, SPEC REFS, INPUT, OUTPUT -- no ACs or ECs
- Detail emerges during PLAN stage (AC-generator, EC-generator)

### Assessment
- Hive Mind's approach is **already aligned** with Anthropic's recommendation
- SPEC defines WHAT, PLAN breaks it into stories, EXECUTE figures out HOW
- The risk of over-specification is lower in Hive Mind because AC/EC generation is deferred

## 5. Evaluation Criteria / Rubric Design

### Anthropic's Approach
- Turned subjective judgments ("is this design good?") into **concrete, gradable terms**
- Four grading dimensions:
  1. **Design Quality** -- penalizes generic AI patterns
  2. **Originality** -- pushes toward creative risk-taking
  3. **Craft** -- technical execution quality
  4. **Functionality** -- does it work?
- Evaluator equipped with **Playwright for e2e testing** -- browser-based verification

### Hive Mind's Current Approach
- **Binary AC/EC evaluation** -- PASS or FAIL, no gradation
- Compliance checking: DONE | MISSING | UNCERTAIN with file:line evidence
- Code review: critical | major | minor severity classification
- Scorecard: letter grade (A/B/C/D/F) based on pass rate percentage
- No subjective quality assessment (design quality, originality, craft)

### Recommendation: Add Multi-Dimensional Scorecard

**Current scorecard is one-dimensional (pass rate).** Could add:
1. **Code quality score** -- from code-reviewer findings (critical/major/minor counts)
2. **Test coverage score** -- from EC pass rates
3. **Regression score** -- from baseline comparison
4. **Compliance score** -- from compliance-reviewer
5. **Architecture score** -- from integration-verifier

**Why this matters:**
- A project can have 100% AC pass rate but terrible code quality
- Multi-dimensional grading surfaces trade-offs the user should know about

## 6. Model-Awareness / Pipeline Simplification

### Anthropic's Evolution
- With Opus 4.6, removed the sprint construct entirely
- Model could natively handle decomposition
- Kept planner and evaluator because they still added value
- Key insight: "the space of interesting harness combinations doesn't shrink as models improve -- it moves"

### Hive Mind's Current Approach
- Fixed pipeline regardless of model capability
- Hardcoded model assignments per agent type (opus/sonnet/haiku)
- No dynamic simplification based on model tier or task complexity

### Recommendation: Adaptive Pipeline Depth

**Short-term:**
- Allow `--simple` flag that skips NORMALIZE, BASELINE, and reduces VERIFY retry count
- For small PRDs (< 500 words), auto-skip NORMALIZE

**Long-term:**
- Complexity estimator at PLAN stage: if plan has < 5 stories, use simplified execution (no waves, no compliance)
- Model-aware retry counts: opus might need fewer verify attempts than haiku

## 7. Parallel Execution

### Anthropic's Approach
- Sequential sprint execution -- one feature per sprint
- No parallel agents mentioned
- Focus on quality per sprint, not throughput

### Hive Mind's Approach
- **Wave-based parallel execution** -- significant advantage
- Non-overlapping stories execute concurrently with bounded concurrency
- Smart file-overlap detection defers conflicting stories to next wave
- Mutex serializes COMMIT sub-stage to prevent git conflicts

### Assessment
- Hive Mind is **ahead** of Anthropic's published harness on parallelism
- This is a genuine differentiator for production use
- No changes needed

---

## 8. Cross-Session Learning

### Anthropic's Approach
- No mention of cross-session learning in either article
- Each pipeline run starts fresh (only handoff artifacts carry state within a run)
- No graduation or knowledge base system

### Hive Mind's Approach
- **Two-tier learning system**: session memory + knowledge base graduation
- Memory grows during execution, graduates to KB based on stability/evidence/generalizability criteria
- Next pipeline run reads KB files and passes to agents as context
- Graduation log provides audit trail

### Assessment
- Hive Mind is **significantly ahead** on learning
- This is a major differentiator -- patterns proven in prior runs inform future runs
- Anthropic's harness has no equivalent

## Summary: What Hive Mind Should Learn

| Area | Priority | Action |
|------|----------|--------|
| Context resets | HIGH | Add stage-boundary and wave-boundary context resets with structured handoff artifacts |
| GAN-inspired iteration | MEDIUM | Replace complex checking layers with simple generator-evaluator loops |
| Playwright e2e testing | MEDIUM | Add browser-based verification as optional evaluator backend |
| Multi-dimensional scorecard | MEDIUM | Expand scorecard beyond pass-rate to include code quality, coverage, regression, compliance |
| Adaptive pipeline | LOW | Allow pipeline simplification for small/simple tasks |
| Subjective quality rubric | LOW | Add design/originality/craft grading for UI-heavy projects |

## What Hive Mind Already Does Better

| Area | Advantage |
|------|-----------|
| Parallel wave execution | Concurrent non-overlapping stories (Anthropic is sequential) |
| Cross-session learning | Graduation system with knowledge base (Anthropic has none) |
| Human-in-the-loop | Checkpoints at every stage (Anthropic is fully autonomous) |
| Binary evaluation | Zero subjectivity bias in AC/EC (Anthropic struggles with self-eval bias) |
| Failure recovery | Multi-level retry with crash recovery and artifact preservation |
| Bug-fix pipeline | Dedicated DIAGNOSE -> FIX -> VERIFY flow |
